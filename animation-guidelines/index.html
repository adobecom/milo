The Browser Rendering Pipeline

Every visual change a browser makes goes through up to four stages, in order. The earlier in the pipeline work is triggered, the more expensive the frame.

01
Style
Recalculate which styles apply to each element.
High cost
02
Layout
Compute the geometry of every affected element.
High cost
03
Paint
Rasterize pixels into layers.
Medium cost
04
Composite
Assemble layers on the GPU.
Low cost

The browser has a 16.7 ms budget per frame to hold 60 FPS (8.3 ms for 120 FPS). Anything that pushes a frame over budget is dropped, which the eye reads as jank.

The rule: animate only properties that stop at the Composite stage. Those are handled on the GPU and never block the main thread. Everything else has a CPU cost that scales with how much of the DOM is affected.

Add Lenis to the Equation

Lenis is a popular JavaScript library that replaces the browser's native scrolling with a smoothed, inertia-based version. It intercepts wheel and touch events and drives the scroll position from a requestAnimationFrame loop on the main thread, every frame, even when nothing is scrolling. During scroll, that loop does real per-frame work competing for the same 16.7 ms (or 8.3 ms at 120 FPS) frame budget as CSS animations and other JavaScript.

Try it: use the Lenis toggle in the corner to enable or disable it. Scroll the page with Lenis on and off, and ideally do it once more with CPU throttling on (see the next section).

The FPS counter often won't move much when Lenis is on. That's not because Lenis is free - it's because Lenis's cost shows up as input lag, not dropped frames. Every wheel tick is intercepted and interpolated over many frames, so the page lags 50-150 ms behind the input even while every frame is drawn on time. With the CPU throttled, that delay becomes a real, felt jitter. Trust the felt experience, not the counter reading.

Try It: Watch the FPS Meter

Each demo animates 500 boxes (each containing a small chunk of text) by changing one CSS property. Use Play all at the top of a category to run every demo in it simultaneously. Real pages rarely animate just one thing, so stacking multiple animations simulates that combined load.

Run the tests with CPU throttling on too. The teams building modern web apps typically work on fast machines; most users don't. A mid-range Windows laptop running antivirus, a videoconferencing app, and a dozen browser tabs has a fraction of that headroom. To simulate it:
  1. Open Chrome DevTools (Cmd+Option+I on Mac, F12 on Windows)
  2. Click the Performance tab
  3. Click the gear icon at the top-right of that panel
  4. Set the CPU dropdown to 4x slowdown (mid-range laptop) or 6x slowdown (low-end)
  5. Re-run the demos and compare the FPS readings
If the page is smooth at 4x slowdown, it'll be smooth for most users. If it isn't, that's the gap that needs to be closed.

Cheap Compositor-only properties

These never leave the GPU. Animate mostly freely, even in continuous and scroll-linked contexts.

transform: translate()
transform: scale()
transform: rotate()
opacity

Medium Paint-triggering properties

These skip layout, but still require rasterizing new pixels. Acceptable in moderation; expensive on element-heavy pages or when larger values are used.

background-color
box-shadow
filter: blur()
border-radius

Expensive Layout-triggering properties

Every frame, the browser has to recompute geometry for the element and every element affected by it. Avoid in continuous or scroll-linked animations.

width
padding
font-size
line-height

Substituting Expensive Properties

Some layout-triggering animations have a clean compositor-only equivalent. Others don't - the compositor substitute is an approximation with real trade-offs to be aware of before reaching for it. The table below splits the two.

True equivalents

These behave the same way the original property does, visually, while staying on the compositor.

Instead of animating…Use…Why it works
top / left / right / bottom transform: translate() Positioned elements don't push siblings either way, so the visual is identical - with no layout cost.
margin for movement transform: translate() Same: the element moves without reflowing siblings. Use only when the surrounding layout doesn't need to react.
background-position parallax transform: translate3d() on the image inside a clipping wrapper Put the image on its own element with overflow: hidden on the parent; translating it gives the same parallax effect on the GPU.
Show / hide via display opacity + visibility (with transition-behavior: allow-discrete) Genuinely composite-friendly. Element keeps its space in layout but is visually removed.

Approximations (read the trade-off)

These cost less on the compositor but they aren't visually identical to the original. The biggest caveat sits in the first row below.

Instead of animating…Approximate with…What's given up
width / height transform: scaleX() / scaleY() Children scale too, including text (which can blur sub-pixels and pixelate icons). Acceptable on a container that's meant to scale uniformly (e.g. a hover-grow card with no text), not on a generic resizable box.
width / height (clipping a region) Animate clip-path (e.g. inset()) Clip-path animations can run on the compositor in modern Chrome when they don't change the element's box. Good for revealing content. The element still occupies its full size in layout.
font-size growth transform: scale() on the text container Same trade-off as above: sub-pixel rendering is interpolated, so the text can look blurry at intermediate scales. Best for short bursts; avoid for long-running scroll-driven growth.
line-height per-line growth No clean substitute. Wrap each visible line in its own element manually, then animate each with transform: translateY(). Per-line typography animation is fundamentally layout work because the browser has to decide where lines break. The split-into-spans workaround is brittle: re-wrapping on resize requires re-splitting.
Sibling-push hover effects transform: scaleX() on the hovered element + overflow: hidden on the parent The hovered element grows visually, but its siblings don't move out of the way - they get clipped. Acceptable when the goal is emphasis, not spatial reorganization.
When in doubt: not every layout animation has a compositor-only equivalent. If the trade-off doesn't fit the use case, the next-best move is usually to reduce the scope of the animation - fewer elements, shorter duration, or skip the animation entirely on lower-end hardware.

Why Windows and Mac Feel Different

The exact same page can run smoothly on a MacBook Air and stutter on a four-year-old Windows laptop. Four factors compound to produce that gap.

1. Hardware diversity

Apple ships ~5 chip variants per year (M1 / M2 / M3 / M4 and their Pro/Max tiers). Even the entry-level MacBook Air has a capable GPU. The hardware floor is high. On Windows, the range is enormous - from gaming laptops with discrete GPUs to corporate laptops from 2019 with Intel UHD integrated graphics.

2. Unified memory on Apple Silicon

On an M-series Mac, CPU and GPU share the same physical RAM. When the browser hands a layer to the GPU, nothing is copied. On a typical Windows laptop, GPU memory is either separate (discrete GPU, requiring PCIe transfers per frame for changed layers) or shared with system RAM through a slower memory path than Apple's unified design. So animations the GPU can "just play" on Mac carry an upload cost on Windows, every frame.

3. The Chrome → GPU path

Both platforms have had GPU-accelerated window compositors for nearly two decades: macOS via Quartz Extreme (2002) and Core Animation (2007), Windows via DWM (Vista, 2006). Both are mature. The practical difference is one level up, in how Chrome talks to the GPU.

On macOS, Chrome uses Apple's Metal API as its graphics backend (default since Chrome 100 in 2022). Metal is low-overhead and well-matched to Apple Silicon. On Windows, Chrome uses Direct3D 11, which carries more API and driver overhead on average - and driver quality varies far more across the wide Windows hardware ecosystem. For WebGL specifically, Chrome on Windows additionally translates calls through ANGLE (OpenGL ES → D3D), though that path only affects WebGL content, not general page rendering.

The net is that the Chrome-to-pixels path on Windows has more layers and more variance than on Mac, on top of the hardware diversity already noted.

4. CPU headroom

Even when an animation can stay on the GPU, pages include scroll handlers, style recalculation, and per-frame JavaScript like Lenis. On an M-series Mac, the CPU has enough headroom to absorb that work well inside the 16.7 ms frame budget. On a four-year-old Windows laptop running antivirus, a videoconferencing app, and 30 browser tabs, the same work blows the budget and frames get dropped.

Net effect: the work is the same on both platforms, but on Mac the Chrome-to-GPU path is shorter, the GPU is closer to memory, the hardware floor is much higher, and the CPU has more headroom. On Windows, all four advantages are weaker - so the same animation that costs nothing on a MacBook Air costs a dropped frame on a four-year-old corporate laptop.

Overall Recommendations

The conclusions are straightforward:

  1. Default to transform and opacity for every animation, on every device. They stay cheap regardless of CPU pressure and scale safely from a current high-end laptop to a four-year-old corporate machine without per-platform tuning.
  2. Be very selective with paint-triggering animations. background-color, box-shadow, filter, and border-radius are cheaper than layout but still cost per-pixel work that scales with surface area. They're fine for short transitions on small elements (a hover state, a button press). Avoid them in continuous, scroll-linked, or page-wide animations.
  3. Don't animate layout-triggering properties. width, height, margin, padding, top / left, font-size, line-height, and the rest of the layout family trigger main-thread work the GPU cannot help with. Their cost compounds with everything else on the page (Lenis, scroll handlers, JS).

If layout animations can't be dropped entirely

The fallback is to feature-gate them on lower-end hardware and let only the machines that can carry the animations actually run them.

This is a fallback, not the goal. The goal is animations that work smoothly for everyone. Anything that can't make that promise should either be rewritten with a compositor-only equivalent or gated so it only runs where the hardware can carry it.
